# AGENTS.md — instructions for AI coding agents

Canonical, tool-neutral instructions for the `billing` service. This repo is old and load-bearing; the
**Landmines** list below is the most important section — read it before editing anything.

## The gates (read before doing anything)

**Ambiguity gate.** Confused or ambiguous? **Stop and ask.** If running unattended, state the ambiguity
and which assumption you chose — never resolve confusion silently.

**Edit gate.** Questions and problem reports get **diagnosis only** until I say *fix* or *implement*.
**Never commit or push.** I author every commit.

**Evidence rule.** A claim about the code that doesn't trace to a **fresh read** is `unverified`. For any
pass/fail claim, show the command you ran and its raw output. This repo has flaky tests — quoting a green
summary you didn't produce is a firing offense.

## Project map

- **What this is:** `billing` — a Django monolith that issues invoices, charges cards via Stripe, and
  emails receipts. Serves ~40M invoice rows.
- **Stack:** Python 3.9 (pinned — see landmines), Django 3.2 LTS, Postgres 13, Celery + Redis, Stripe.
- **How to run it:** `make up` (docker-compose: web, worker, db, redis).
- **How to run tests:** `make test` (pytest-django). Some payment tests are network-flaky; re-run once
  before reporting a failure, and never claim green without the raw output.
- **How to lint/format:** `make lint` (flake8 + black --check).
- **Domain terms:** *invoice* = a finalized charge document (immutable once issued); *draft* = a
  pre-issue invoice; *dunning* = the retry sequence for failed payments.
- **Landmines:**
  - `billing/migrations/` — generated by Django; never hand-edit.
  - `billing/invoice_pdf/` — external partners parse these PDFs byte-for-byte. Layout changes are a
    contract break, not a cosmetic tweak. Do not touch without a decision entry.
  - `billing/stripe/webhooks.py` — the signature-verification block looks like boilerplate you can drop.
    It is not. Removing it accepts forged webhooks.
  - Python is pinned to 3.9 by a C-extension dependency (`weasyprint` toolchain). Upgrading is a project,
    not a line change.

## Working rules

- **One choke point per dependency.** All Stripe calls go through `billing/stripe/client.py`. Don't call
  the Stripe SDK directly from views or tasks.
- **Deterministic core.** Money math lives in `billing/money.py` (integer minor units, tested). Never do
  arithmetic on amounts inline.
- **Stay in scope.** Touch only the files named in the task. No "while I'm here" refactors in a
  five-year-old codebase.

## The ledgers (started when we adopted Keel — forward only, not backfilled)

- **`docs/decisions.md`** — every deliberate choice from adoption onward, dated, with a "revisit if."
- **`docs/falsified.md`** — every killed idea, with the mechanism and the numbers. Check it before
  proposing a migration; several big ones are already dead.

## Reporting format

Verdict first, then metrics against a named baseline, then caveats, then recommendation last.
