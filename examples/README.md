# Examples

Two worked examples of Keel **filled in** — not blank templates (the files at the repo root are those),
but realistic snapshots of what the ledgers, `AGENTS.md`, and roles look like after a few weeks of real
use. Read the `README.md` in each folder first; it narrates the story the files tell.

| Example | Scenario | Shows |
|---|---|---|
| [`greenfield-linkli/`](greenfield-linkli/) | A new URL-shortener API, Keel from day one | The kernel filled from scratch · a **proposed → superseded** decision after review · a killed idea routed to `falsified.md` |
| [`brownfield-billing/`](brownfield-billing/) | Keel retrofitted onto a legacy Django billing service | Fill-*forward* (never backfill) · landmines in the project map · introducing the reviewer, then the auditor |

Both examples deliberately reuse the same ledger formats and role prompts as the root kit — the point is
to see them *carrying real content*, especially the [proposed-vs-ratified](../docs/guide/04-ledgers.md#who-owns-an-entry-proposed-vs-ratified)
boundary in action.
